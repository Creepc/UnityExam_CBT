1. GameManager와 Singleton Pattern

GameManager는 점수, 게임 상태, 스테이지 진행, 게임 오버 여부처럼 게임 전체에서 공유되는 상태를 관리하는 스크립트다. 이런 관리 객체가 여러 개 생기면 점수가 중복 계산되거나 게임 상태가 서로 다르게 저장될 수 있으므로 보통 하나만 존재하게 만든다.

싱글턴 패턴은 클래스 안에 자기 자신을 가리키는 static Instance를 두고, 다른 스크립트가 GameManager.Instance 형태로 접근하게 하는 방식이다. static 변수는 각 오브젝트마다 따로 생기는 값이 아니라 클래스 자체에 속하는 값이다.

  • Awake()에서 이미 Instance가 있으면 새로 생긴 GameManager를 삭제해 중복을 막는다.
  • 씬이 바뀌어도 유지해야 하면 DontDestroyOnLoad(gameObject)를 함께 쓴다.
  • 시험에서는 “하나만 존재해야 하는 관리자 객체를 전역 접근하기 위해 사용”이라고 쓰면 안전하다.

답안 문장: 싱글턴은 하나의 인스턴스만 생성되도록 제한하고 static Instance를 통해 어디서든 접근하게 하는 패턴이다.

2. Unity Lighting: Baked, Realtime, Mixed

Baked Lighting은 빛과 그림자 결과를 실행 전에 미리 계산해서 라이트맵에 저장하는 방식이다. 벽, 바닥, 건물처럼 움직이지 않는 정적 오브젝트에 적합하고, 실행 중 조명 계산이 줄어 성능에 유리하다.

Realtime Lighting은 실행 중 매 프레임 조명과 그림자 변화를 계산한다. 움직이는 캐릭터, 이동하는 물체, 회전하는 조명, 손전등처럼 실시간 변화가 필요한 경우에 사용한다. 대신 계산 비용이 크다.

Mixed Lighting은 정적인 부분은 Baked로 처리하고 동적인 부분은 Realtime 영향을 받도록 섞는 방식이다. Baked 환경에서도 움직이는 캐릭터가 주변 조명과 어울리게 하려면 Light Probe를 사용해 주변의 구워진 조명 정보를 샘플링하게 할 수 있다.

  • Baked: 정적 오브젝트, 미리 계산, 성능 좋음.
  • Realtime: 동적 오브젝트, 실시간 그림자, 비용 큼.
  • Light Probe: Baked 조명 환경에서 동적 오브젝트가 주변 간접광을 받게 보완.

함정 포인트: Baked는 움직이는 오브젝트의 실시간 그림자 변화를 자동으로 완벽히 반영하지 않는다.

3. Animator Controller Layer

Animator Layer는 애니메이션을 여러 층으로 나누어 조합하는 기능이다. 예를 들어 하체는 달리기 애니메이션을 유지하고, 상체만 공격 애니메이션으로 바꾸고 싶을 때 사용한다.

Layer에는 Weight가 있다. Weight가 0이면 영향이 없고, 1에 가까울수록 해당 레이어가 최종 포즈에 강하게 반영된다. Blend Mode가 Override이면 이전 레이어의 포즈를 덮어쓸 수 있고, Additive이면 기존 포즈에 변화량을 더하는 방식이다.

Avatar Mask는 특정 신체 부위에만 레이어를 적용할 때 사용한다. 상체 공격 + 하체 달리기처럼 부분 애니메이션을 만들 때 중요하다.

답안 문장: Animator Layer는 여러 애니메이션을 층으로 조합하며, Weight와 Override/Additive 설정, Avatar Mask에 따라 최종 애니메이션이 결정된다.

4. Cinemachine

Cinemachine은 카메라 연출을 쉽게 만들기 위한 Unity 패키지다. 실제 화면을 렌더링하는 것은 Main Camera이고, Main Camera에 Cinemachine Brain 컴포넌트를 붙인다.

Virtual Camera는 실제 렌더링을 하지 않는다. 대신 카메라 위치, Follow 대상, Look At 대상, 렌즈 정보 같은 “카메라 설정”을 가진다. 씬 안에 여러 Virtual Camera를 두고 상황에 따라 전환할 수 있다.

  • Cinemachine Brain: Main Camera에 붙어서 어떤 Virtual Camera를 사용할지 결정한다.
  • Virtual Camera: 실제 카메라가 아니라 카메라 설정 정보다.
  • Priority: 여러 Virtual Camera 중 선택 우선순위에 영향을 준다.
  • Blend: 카메라 전환을 부드럽게 연결한다.

함정 포인트: Virtual Camera가 직접 렌더링하는 것이 아니라 Brain이 Main Camera를 제어한다.

5. VR, AR, MR, XR

VR은 Virtual Reality, 가상현실이다. 사용자가 현실을 거의 보지 않고 완전히 가상의 공간 안에서 가상 객체와 상호작용한다.

AR은 Augmented Reality, 증강현실이다. 현실 공간이나 카메라 화면 위에 가상 정보, 이미지, 캐릭터, 안내 표시를 겹쳐 보여준다.

MR은 Mixed Reality, 혼합현실이다. 현실 공간을 인식하고 현실 객체와 가상 객체가 서로 영향을 주거나 상호작용한다. 예를 들어 가상 공이 실제 책상 위에 놓이거나 실제 벽 뒤로 가려지는 상황이다.

XR은 Extended Reality, 확장현실이다. VR, AR, MR을 모두 포함하는 상위 개념이다.

암기 문장: 가상 공간만 보면 VR, 현실 위에 가상을 겹치면 AR, 현실과 가상이 상호작용하면 MR, 전부 묶으면 XR.

6. Object Pooling

Object Pooling은 오브젝트를 필요할 때마다 새로 생성하고 삭제하지 않고, 미리 만들어 둔 오브젝트를 재사용하는 최적화 기법이다. “오브젝트 쿨링”이 아니라 “오브젝트 풀링”이 정확한 용어다.

Unity에서 Instantiate()와 Destroy()를 반복하면 메모리 할당과 해제가 자주 일어나고, Garbage Collection 때문에 순간적인 프레임 드랍이 생길 수 있다. 그래서 총알, 발판, 몬스터, 이펙트, 코인처럼 자주 생기고 사라지는 오브젝트는 풀링을 사용한다.

원리는 간단하다. 시작할 때 필요한 수만큼 만들어 풀에 넣어두고, 필요하면 SetActive(true)로 켠다. 사용이 끝나면 삭제하지 않고 SetActive(false)로 끈 뒤 다시 풀에 돌려놓는다.

답안 문장: Object Pooling은 반복 생성/삭제 비용과 GC 부담을 줄이기 위해 오브젝트를 미리 생성하고 활성화/비활성화로 재사용하는 기법이다.

7. UI 렌더링 순서

Unity UGUI에서 같은 Canvas 안의 Text, Image, Button 같은 UI 요소는 보통 카메라와의 물리적인 Z축 거리보다 Hierarchy의 Sibling 순서가 중요하다. 같은 부모 아래에서 아래쪽에 있는 UI가 나중에 그려져 더 앞에 보인다.

예를 들어 Canvas 아래에 Background, Button, Text 순서로 있으면 Background가 먼저 그려지고 Text가 가장 나중에 그려져 앞에 보인다. UI가 뒤에 가려지거나 앞뒤가 이상하면 먼저 Hierarchy 순서를 확인해야 한다.

다만 Canvas가 여러 개라면 Canvas의 Sort Order, Sorting Layer, Order in Layer 같은 설정이 중요해진다. World Space Canvas나 3D 오브젝트는 카메라 거리나 깊이 테스트가 영향을 줄 수 있다.

함정 포인트: 기본 Screen Space Overlay UI는 “Z축 거리만으로 앞뒤가 결정된다”가 아니다. 같은 Canvas 안에서는 Sibling 순서를 봐야 한다.

8. Blend Tree

Blend Tree는 Animator에서 여러 애니메이션 클립을 파라미터 값에 따라 부드럽게 섞어 재생하는 기능이다. 단순 Transition은 상태가 바뀌는 느낌이 강하지만, Blend Tree는 중간 값을 이용해 자연스럽게 연결한다.

1D Blend Tree는 Speed 같은 파라미터 하나를 사용한다. Speed가 0이면 Idle, 0.5면 Walk, 1이면 Run처럼 설정할 수 있고, 중간 값에서는 애니메이션이 섞인다.

2D Blend Tree는 Horizontal과 Vertical처럼 두 파라미터를 사용해 앞, 뒤, 좌, 우, 대각선 이동 애니메이션을 자연스럽게 섞을 수 있다.

답안 문장: Blend Tree는 Speed나 Direction 같은 파라미터 값에 따라 정지, 걷기, 뛰기, 방향 이동 애니메이션을 자연스럽게 혼합하는 기능이다.

9. Rigidbody와 MovePosition

Rigidbody는 Unity 물리 엔진의 영향을 받게 하는 컴포넌트다. Rigidbody가 있으면 중력, 충돌, 힘, 속도 같은 물리 효과를 사용할 수 있다.

Rigidbody가 붙은 오브젝트를 transform.position으로 직접 바꾸면 물리 엔진을 통한 이동이 아니라 위치를 강제로 바꾸는 방식이 된다. 이 경우 충돌 계산이 누락되거나 벽을 뚫는 현상이 생길 수 있다.

물리 기반 이동은 보통 FixedUpdate()에서 Rigidbody.MovePosition(), velocity, AddForce 등을 사용한다. 특히 MovePosition은 Rigidbody의 물리 이동을 처리할 때 자주 쓰인다.

  • Update(): 프레임마다 호출.
  • FixedUpdate(): 물리 연산 주기에 맞춰 호출.
  • MovePosition(): Rigidbody를 물리 시스템과 맞춰 위치 이동.

함정 포인트: Rigidbody가 있어도 transform.position으로 계속 강제 이동하면 충돌이 항상 안전하게 처리되는 것이 아니다.
